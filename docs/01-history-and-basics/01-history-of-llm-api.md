# LLM APIの経緯

## この教材で身につくこと

- LLM APIというビジネスモデルが生まれた背景
- GPT-3公開からエージェント時代までの進化の流れ
- なぜ多くの企業が自社でモデルを持たずAPIを使うのか

## 概要

大規模言語モデル（LLM）は数十億〜数兆パラメータを持ち、推論だけでも
大規模なGPUクラスタを必要とします。個々の企業が自前で運用するのは
非現実的なため、「モデルをAPI経由のサービスとして提供する」形態が
2020年前後から主流になりました。

## 位置づけ

このチュートリアル全体の出発点です。ここで得た歴史的文脈は、
標準化（04章）で扱う「なぜ仕様がベンダーごとに異なるのか」を
理解する助けになります。

## 仕組み解説

### 進化のタイムライン

| 時期 | できごと |
|------|----------|
| 2018〜2019 | Transformerベースのモデル（BERT等）が研究分野で普及 |
| 2020 | OpenAIがGPT-3をAPI限定で公開。「モデルは秘匿、利用は従量課金」の型が定着 |
| 2022 | ChatGPT公開。対話形式（system/user/assistant）のAPI設計が広がる |
| 2023 | AnthropicがClaude APIを公開。ツール呼び出し（Function Calling）が普及 |
| 2024 | MCP（Model Context Protocol）公開。ツール接続の標準化が進む |
| 2025〜 | 拡張思考・エージェント基盤（Managed Agents等）へと機能が拡大 |

### なぜ「API化」が主流になったか

| 理由 | 説明 |
|------|------|
| 計算資源 | 数百GB級のモデル重みと専用GPUクラスタが必要 |
| 運用コスト | モデル更新・スケーリング・障害対応をベンダー側に集約できる |
| 知的財産保護 | モデル重み自体を公開せずに機能だけを提供できる |
| 従量課金 | 使った分だけ支払う形にでき、初期投資が不要 |

### 機能拡張の方向性

単純なテキスト補完から、以下の順で機能が拡張されてきました。

```
テキスト補完
  → チャット形式（役割ベースのメッセージ配列）
    → ツール呼び出し（外部関数・APIとの連携）
      → マルチモーダル（画像・PDF入力）
        → 拡張思考（内部推論の可視化）
          → エージェント基盤（自律的なタスク遂行）
```

チャット形式は02章、ツール呼び出しは02章・05章、マルチモーダルと拡張思考は
03章、エージェント基盤は05章（と04章のMCP）で、それぞれ詳しく扱います。

### 主要プロバイダの独立公開時期

LLM APIは1社の独占ではなく、複数の企業がそれぞれ独自にAPIを公開しています。

| プロバイダ | API公開時期 | 提供元 |
|-----------|-------------|--------|
| OpenAI | 2020年（GPT-3） | OpenAI |
| Anthropic | 2023年（Claude API） | Anthropic |
| Google | 2023年（Gemini API） | Google |

以降の章では、特に**OpenAI公式API**と**Claude API**を軸に、
同じ概念がベンダーごとにどう異なる形で表現されるかを比較していきます。

## 実装例

初期のテキスト補完APIは以下のような単純な構造でした（概念例）。

```json
{
  "model": "text-davinci-003",
  "prompt": "以下の文章を要約してください: ...",
  "max_tokens": 200
}
```

現在の対話形式APIでは、役割付きメッセージの配列を送ります。

```json
{
  "model": "claude-opus-4-8",
  "max_tokens": 1024,
  "messages": [
    {"role": "user", "content": "以下の文章を要約してください: ..."}
  ]
}
```
