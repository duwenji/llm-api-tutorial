# トークンとコンテキスト

## この教材で身につくこと

- トークン化の仕組みと課金・制限への影響
- コンテキストウィンドウの意味と実務上の制約
- `max_tokens`と出力打ち切りの関係

## 概要

LLMは文字列をそのまま扱うのではなく、「トークン」という単位に
分割してから処理します。料金・制限は基本的にトークン数で決まります。

## 位置づけ

03章のプロンプトキャッシュ、05章のコスト最適化を理解するための
前提知識です。

## 仕組み解説

### トークン化とは

トークンは単語より小さい単位（サブワード）に分割されることが多く、
英語では1トークン≒0.75単語、日本語では1文字が複数トークンに
なる場合もあります。モデルごとにトークナイザーが異なります。

| 誤解しがちな点 | 正しい理解 |
|----------------|-----------|
| 1文字=1トークン | モデル・言語によって比率が変わる |
| 他社モデルのトークナイザーで概算できる | ベンダー間で流用不可（要`count_tokens`） |

### コンテキストウィンドウ

コンテキストウィンドウとは、1リクエストで扱える
「入力+出力」トークン数の上限です。

```
コンテキストウィンドウ = 入力トークン + 出力トークン（max_tokens）
```

例えばコンテキストウィンドウが1,000,000トークンのモデルでも、
入力が90万トークンなら出力に使える枠は10万トークンに制限されます。

### max_tokensと打ち切り

`max_tokens`に達すると、応答は途中で打ち切られ
`stop_reason: "max_tokens"`が返ります。長文出力にはストリーミングを使い、
十分な`max_tokens`を確保します。

```json
// ❌ 悪い例: max_tokensが小さすぎて出力が途中で切れる
{"model": "claude-opus-4-8", "max_tokens": 50, "messages": [...]}
```

```json
// ✅ 良い例: 長文生成には十分なmax_tokensとストリーミングを使う
{"model": "claude-opus-4-8", "max_tokens": 8000, "stream": true, "messages": [...]}
```

## 実装例

```python
# トークン数を事前に確認する（例: Anthropic API）
count = client.messages.count_tokens(
    model="claude-opus-4-8",
    messages=[{"role": "user", "content": "見積もりたい本文..."}],
)
print(count.input_tokens)
```

```python
# コスト概算（例: 入力 $3/1M, 出力 $15/1M）
input_cost = count.input_tokens * 3 / 1_000_000
print(f"推定入力コスト: ${input_cost:.4f}")
```
